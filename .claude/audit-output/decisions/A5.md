# Decision A5 — `server.ts` split deferred / re-scoped

**Status:** closed without code change
**Date:** 2026-05-13
**Sprint:** 4
**Original audit recommendation:** split `src/lib/supabase/server.ts` into a
service-role client (`admin.ts`) and a user-scoped SSR client (`ssr.ts`), update
~40 imports, add RLS regression tests.

## Why we are not splitting

Pulse runs in **single-user mode** (see CLAUDE.md and the `feedback_scope_single_user`
memory). Concretely:

- `PULSE_USER_ID` is a hardcoded environment variable that identifies the only
  legitimate owner of all rows.
- `createClient()` in `src/lib/supabase/server.ts` already **is** the admin
  client (`createAdminClient()`), with `auth.getUser()` monkey-patched to return
  the configured owner ID (see `patchAuthGetUser()` introduced in D3 part 4).

The audit's split assumes a multi-user app where service-role and user-scoped
clients must be separated to enforce least-privilege at the SQL layer. In our
case there is no second user to scope away from, so the split has no security
substance — it would only be cosmetic.

## What the security substance of A5 already covers

The defenses A5 wanted are present, just in different files:

| A5 concern                                  | Where it lives now |
| ------------------------------------------- | ------------------ |
| `auth.getUser()` patch isolation            | `patchAuthGetUser()` helper in `server.ts` (D3 part 4) |
| Verify RLS policies are enabled and correct | `supabase/migrations/*_rls_*` and A6 verification (sprint 1) |
| `auth.users` trigger creates `user_settings` | G4 (sprint 2) |
| Service-role boundary documented            | JSDoc on `createAdminClient()` in `src/lib/supabase/admin.ts` |

If Pulse ever onboards a second user the RLS policies will keep their data
isolated even though every server route currently uses the admin key — Supabase
RLS is the actual enforcement layer, not the choice of client.

## What we could still do (low priority follow-ups)

These are not blockers; capture for future reference.

1. **Rename `createClient` → `createServerClient`** to signal that the returned
   client bypasses RLS. Touches ~40 importers. Cosmetic.
2. **`assertOwner(client)` helper** that throws if `auth.getUser()` does not
   match `PULSE_USER_ID`. Defensive belt-and-braces for routes that take a
   user-id from URL/query params.
3. **RLS smoke test** (vitest + Supabase test schema) that inserts two users,
   queries with one, and asserts the other's rows are unreachable. Pure
   regression coverage.

All three are deferred until Pulse genuinely needs multi-tenancy or the
defense-in-depth becomes worth the upkeep.
