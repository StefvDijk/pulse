# Competitor Teardown Template

Use this exact structure for every teardown. One file per app at `research/teardowns/<app-slug>.md`.

```markdown
# <App Name> — Teardown

**Researched:** YYYY-MM-DD
**Sources:** [URLs to official site, App Store listing, key reviews, blog posts]

## 1. Snapshot
- **Target user:** (e.g. "endurance runners training for race", "general health-conscious Apple Watch owners")
- **Platform:** iOS / Android / Web
- **Pricing:** free / $X/mo / $Y/yr / lifetime
- **Maturity:** founded year, est. user base if known
- **Data sources it consumes:** Apple Health / Garmin / Strava / manual / proprietary device

## 2. Jobs-to-be-done
What does the user "hire" this app to do? 3-5 bullets, phrased as outcomes not features.

## 3. Killer features (top 3-5)
The features that show up in every review/marketing as the differentiator. Brief description each.

## 4. Daily-loop screen
Which single screen is designed to be opened every morning? What's on it?

## 5. Onboarding & first-week experience
- How long to first value?
- What data does it ask for?
- How does it handle empty state?

## 6. Insights / AI layer
Does it derive insights, or just visualize data? If insights: rules-based or ML or LLM? Examples of insight phrasing.

## 7. Retention mechanism
What brings the user back daily? Streak, score, notification, social, content?

## 8. Pricing & business model
Free tier limits, paid tier features, subscription vs lifetime.

## 9. Design language (1-2 sentences)
Dark/light, minimal/dense, chart-heavy/copy-heavy.

## 10. What Pulse already does as well or better
Honest list. Where we already match or exceed.

## 11. What Pulse is missing vs this app
Specific feature gaps. This is the gold for the gap matrix.

## 12. What this app does badly (anti-patterns to avoid)
Common complaints in reviews, friction points.
```
